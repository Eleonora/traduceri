Everyday GIT With 20 Commands Or So

De comenzile [Basic Repository] se folosesc oamenii care deţin un repozitoriu --- ceea ce înseamnă toţi, deoarece fiecare ramificare al git este un repozitoriu.

în afară de aceasta, sînt esenţiale comenzile [Individual Developer (Standalone)] pentru oricine care face un commit(transmitere), chiar şi pentru cineva care lucrează singur.

Dacă lucraţi cu alţii oameni, veţi avea nevoie de comenzi care sint listate deasemenea in secţiunea [Individual Developer (Participant)].

Oamenii care joacă rolul [Integrator]-ului au nevoie să înveţe cîteva comenzi în plus la cele de mai sus.

Comenzile[Repository Administration] sînt pentru administratorii de sistemă care sînt responsabili pentru păstrarea şi transmiterea repozitoriilor git.

Basic Repository

Toţi folosesc această comandă pentru a deservi repozitoriile git.

  • git-init(1) sau git-clone(1) pentru a crea un repozitoriu nou.

  • git-fsck(1) pentru a căuta repozitoriu dacă conţine erori.

  • git-gc(1) pentru a face lucrul obişnuit cum ar fi reîmpachetarea şi micşorarea.

Exemple

Check health and remove cruft. 
$ git fsck (1)
$ git count-objects (2)
$ git gc (3)

  1. rulînd fără --full este deobicei mai ieftin şi asigură repozitorului "sănătate" destul de bună.

  2. vedeţi acum cîte obiecte libere sînt şi cît spaţiu pe disc este pierdut fără reîmpachetare.

  3. reîmpachetează repozitoriul local şi rulează alte funcţii de rutină. Rulînd fără --prune este o operaţiă sigură chiar altele sînt derulare.

Reîmpachetează un mic proiect intr-un singur pachet. 
$ git gc (1)
$ git gc --prune

  1. împachetează toate obiectele disponibile din refs intr-un singur pachet, apoi şterge alte pachete.

Individual Developer (Standalone)

Un developer individual nu face schimb de patch-uri cu alţii, şi lucrează singur într-un singur repozitoriu, folosind următoarele comenzi.

  • git-show-branch(1) pentru a vedea unde vă aflaţi.

  • git-log(1) pentru a vedea ce s-a întîmplat.

  • git-checkout(1) şi git-branch(1) pentru a schimba ramificarea.

  • git-add(1) pentru a lucra cu file-ul index.

  • git-diff(1) şi git-status(1) pentru a vedea în ce punct vă aflaţi cu lucrul.

  • git-commit(1) pentru a vă muta la ramura curentă.

  • git-reset(1) şi git-checkout(1) (cu parametrii căii de acces) pentru a anula schimbările.

  • git-merge(1) pentru a uni între ramuri locale.

  • git-rebase(1) pentru a deservi ramurile temei.

  • git-tag(1) pentru a marca punct cunoscut.

Exemple

Folosiţi tarball ca punct de start pentru un repozitoriu nou. 
$ tar zxf frotz.tar.gz
$ cd frotz
$ git-init
$ git add . (1)
$ git commit -m "importă copacul sursă al frotz."
$ git tag v2.43 (2)

  1. adăugaţi totul sub directoria curenta.

  2. faceţi un lightweight, tagurilor neanotate.

Creaţi o ramura a topic-ului şi dezvoltaţi. 
$ git checkout -b alsa-audio (1)
$ edit/compile/test
$ git checkout -- curses/ux_audio_oss.c (2)
$ git add curses/ux_audio_alsa.c (3)
$ edit/compile/test
$ git diff HEAD (4)
$ git commit -a -s (5)
$ edit/compile/test
$ git reset --soft HEAD^ (6)
$ edit/compile/test
$ git diff ORIG_HEAD (7)
$ git commit -a -c ORIG_HEAD (8)
$ git checkout master (9)
$ git merge alsa-audio (10)
$ git log --since='3 days ago' (11)
$ git log v2.43.. curses/ (12)

  1. crează o ramură nouă a topicului.

  2. intoarceţi schimbările voastre stricate in curses/ux_audio_oss.c.

  3. trebuie să spuneţi git-ului dacă aţi adaugat un file nou; ştergerile şi modificările vor fi inregistrate dacă rulaţi comanda git commit -a mai tîrziu.

  4. ca să vedeţi ce schimbări trimiteţi.

  5. trimiteţi orice ce voi aţi testat spre sfîrşit.

  6. luaţi ultima trimitere înapoi, lăsînd ce este în ramificarea lucrătoare.

  7. look at the changes since the premature commit we took back.

  8. refaceţi commit-ul distrus înainte, folosind mesajul care l-aţi scris iniţial.

  9. schimbaţi la ramura principală.

  10. adaugaţi o ramură topic ramura DVS principală.

  11. revedeţi log-urile trmiterilor (commit); other forms to limit output can be combined and include --max-count=10 (show 10 commits), --until=2005-12-10, etc.

  12. vedeţi numai schimbările care au atins ce e în directoriulcurses/, de la v2.43.

Developer Individual (Participant)

Un developer lucrînd ca participant într-un grup al unui proiect trebuie să înveţe cum să comunice cu alţii, şi foloseşte aceste comenzi în adăugare la cele necesare unui developer desinestătător.

  • git-clone(1) menţinînd fluxul către principalul repozitoriu.

  • git-pull(1) and git-fetch(1) de la "origine" pentru a menţine la curent cu un flux.

  • git-push(1) to shared repository, if you adopt CVS style shared repository workflow.

  • git-format-patch(1) to prepare e-mail submission, if you adopt Linux kernel-style public forum workflow.

Exemple

Clone the upstream and work on it. Feed changes to upstream. 
$ git clone git://git.kernel.org/pub/scm/.../torvalds/linux-2.6 my2.6
$ cd my2.6
$ edit/compile/test; git commit -a -s (1)
$ git format-patch origin (2)
$ git pull (3)
$ git log -p ORIG_HEAD.. arch/i386 include/asm-i386 (4)
$ git pull git://git.kernel.org/pub/.../jgarzik/libata-dev.git ALL (5)
$ git reset --hard ORIG_HEAD (6)
$ git gc --prune (7)
$ git fetch --tags (8)

  1. repetaţi in caz că e nevoie.

  2. extrageţi patch-uri din ramura DVS pentru submisii e-mail.

  3. git pull întărire de la origine implicit şi uneşte în ramura curentă.

  4. imediat după scoatere, vedeţi schimbările făcute upstream începînd cu ultima data cînd aţi verificat, numai în zona de care suntem interesaţi.

  5. aduce dintr-o ramură anume a unui anume repozitoriu şi uneşte.

  6. reîntoarce scoaterea.

  7. garbage collect leftover objects from reverted pull.

  8. din cînd în cînd, scoateţi tag-uri oficiale de la origine şi salvaţi-le sub .git/refs/tags/.

$ git checkout master

  git am -3 -k (2)

  1. crează o ramură privată bazată pe bine cunoscutul tag.

  2. forward port toate schimbările din ramura private2.6.14, în ramura master fără o "fuziune" formală.

Integrator

O persoană activează ca integrator într-un proiect şi primeşte schimbările făcute de alţii, le revede şi integrează şi publică rezultatul pentru alţii pentru folosire, folosid aceste comenzi împreună cu acele ce sînt cerute de alţi participanţi.

  • git-am(1) pentru a aplica patch-uri trimise prin e-mail de la contributori.

  • git-pull(1) to merge from your trusted lieutenants.

  • git-format-patch(1) to prepare and send suggested alternative to contributors.

  • git-revert(1) to undo botched commits.

  • git-push(1) to publish the bleeding edge.

Examples

My typical GIT day. 
$ git status (1)
$ git show-branch (2)
$ mailx (3)
& s 2 3 4 5 ./+to-apply
& s 7 8 ./+hold-linus
& q
$ git checkout -b topic/one master
$ git am -3 -i -s -u ./+to-apply (4)
$ compile/test
$ git checkout -b hold/linus && git am -3 -i -s -u ./+hold-linus (5)
$ git checkout topic/one && git rebase master (6)
$ git checkout pu && git reset --hard next (7)
$ git merge topic/one topic/two && git merge hold/linus (8)
$ git checkout maint
$ git cherry-pick master~4 (9)
$ compile/test
$ git tag -s -m "GIT 0.99.9x" v0.99.9x (10)
$ git fetch ko && git show-branch master maint 'tags/ko-*' (11)
$ git push ko (12)
$ git push ko v0.99.9x (13)

  1. vezi ce făceam eu, dacă făceam ceva.

  2. vezi ce ramuri topic am eu şi cît de pregătite sunt ele.

  3. citeşte mail-uri, salvează cele ce sunt aplicabile, şi salvează altele care nu sînt chiar pregătite.

  4. aplică-le, interactiv, cu sign-offs.

  5. crează ramură topic cum se cere şi aplică, iarăşi cu sign-offs.

  6. rebase internal topic branch care nu a fost fuzionată cu ramura master, sau expuse ca parte a unei ramure stabile.

  7. restart pu de fiecare dată de la următorul.

  8. şi teancul de ramuri topic încă în gătire.

  9. backport a critical fix.

  10. create a signed tag.

  11. mă asigur că nu am întors accidental ramura master dincolo de ce deja am împins pe GIT. ko stenografiază punctele în repozitoriul meu la kernel.org, şi arată cam aşa: 

    $ cat .git/remotes/ko
URL: kernel.org:/pub/scm/git/git.git
Pull: master:refs/tags/ko-master
Pull: next:refs/tags/ko-next
Pull: maint:refs/tags/ko-maint
Push: master
Push: next
Push: +pu
Push: maint

    Spre afişare din git show-branch, master ar trebui să aibă orice ko-master are, şi next ar trebui să aibă orice ko-next are.

  12. push out the bleeding edge.

  13. push the tag out, too.

Repository Administration

Administrator al repozitoriului foloseşte urmăatoarele unelte pentru a seta şi a menţine accesul la repozitoriu de către developeri.

  • git-daemon(1) pentru a lăsa anonimii să descarce din repozitoriu.

  • git-shell(1) poate fi folosit ca restricted login shell pentru a împărţi repozitoriul central cu alţi utilizatori.

update hook howto are un bun exemplu de a conduce un repozitoriu central.

Example

Noi presupunem următoarele în /etc/services 
$ grep 9418 /etc/services
git             9418/tcp                # Git Version Control System
Rulaţi git-daemon pentru serivrea /pub/scm from inetd. 
$ grep git /etc/inetd.conf
git     stream  tcp     nowait  nobody \
  /usr/bin/git-daemon git-daemon --inetd --export-all /pub/scm

Linia actuală de configurare ar trebui să fie una.

Rulaţi git-daemon pentru servirea /pub/scm din xinetd. 
$ cat /etc/xinetd.d/git-daemon
# default: off
# description: Serverul git oferă acces la repozitoriile git
service git
{
        disable = no
        type            = UNLISTED
        port            = 9418
        socket_type     = stream
        wait            = no
        user            = nobody
        server          = /usr/bin/git-daemon
        server_args     = --inetd --export-all --base-path=/pub/scm
        log_on_failure  += USERID
}

Verificaţi documentarea xinetd(8) şi instalarea, acesta este din sistemul Fedora. Altele ar putea fi diferite.

Daţi acces push/pull only developerilor. 
$ grep git /etc/passwd (1)
alice:x:1000:1000::/home/alice:/usr/bin/git-shell
bob:x:1001:1001::/home/bob:/usr/bin/git-shell
cindy:x:1002:1002::/home/cindy:/usr/bin/git-shell
david:x:1003:1003::/home/david:/usr/bin/git-shell
$ grep git /etc/shells (2)
/usr/bin/git-shell

  1. log-in shell e setat la /usr/bin/git-shell, care nu permite nimic în afară de git push şi git pull. Utilizatorul ar trebui să ia un ssh acces la maşină.

  2. în multe distribuţii /etc/shells are nevoie să listeze ce este folosit ca shell pentru login.
